# 0.5 importLoaders 属性 - 3.3 的一些内容补充

当 css 文件中引入了另一个 css 文件时，被引入的文件并不会进行代码兼容化处理。这时候就要设置 importLoaders

const { resolve } = require('path')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'main.js',
    path: resolve(__dirname, 'dist')
  },
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [
          'style-loader', 
          {
            loader: 'css-loader',
            options: {
              /**
               * postcss-loader 将处理好的 css 给 css-loader 后，
               * css-loader 发现其提交的 css 有引入另一个 css，
               * 这时候会往前一步，即 importLoaders: 1
               */
              importLoaders: 1
            }
          },
          'postcss-loader' // 会自动找 postcss.config.js 配置文件
        ]
      },
      {
        test: /\.less$/,
        use: [
          'style-loader', 
          'css-loader',
          'postcss-loader', // 会自动找 postcss.config.js 配置文件
          'less-loader'
        ]
      }
    ]
  }
}

# 0.8 使用 url-loader 处理图片

以 base64 格式打包图片，减少请求次数。

url-loader 内部其实也可以调用 file-loader

  1. 下载：npm i -D url-loader
  2. webpack.config.js：
{
  test: /\.(png|svg|gif|jpe?g)$/,
    use: [
      {
        loader: 'url-loader',
        options: {
          name: 'img/[name].[hash:6].[ext]',
          limit: 25 * 1024 // 25kb 以下才转 base64
        }
      }
    ]
}

# 0.9 asset 处理图片

asset module type：

  1. asset/resource：相当于 file-loader，将资源拷贝到指定的目录，分开请求
  2. asset/inline：相当于 url-loader，导出一个资源的 data URI，以 base64 格式打包图片，减少请求次数
  3. asset/source：相当于 raw-loader，将文件导入为字符串
  4. asset：在导出一个 data URI（base64 的图片） 和发送一个单独的文件之间 自动选择

file-loader 替代写法：

  1. 全局配置：webpack.config.js：（这样的话字体资源也会被打包到 img 目录下面，不大合适）
output: {
  filename: 'main.js',
  path: resolve(__dirname, 'dist'),
  assetModuleFilename: 'img/[name].[hash:4][ext]' // [hash:4].[ext] 中间那个点不需要写了，会自动加的
}

{
  test: /\.(png|svg|gif|jpe?g)$/,
  type: 'asset/resource'
}

# 1.webpack 简介

# 1.1 webpack 是什么

  1. webpack 是一种前端资源构建工具，一个静态模块打包器（module bundler）。在 webpack 看来，前端的所有资源文件（js/json/css/img/less/...）都会作为模块处理，它将根据模块的依赖关系进行静态分析，打包生成对应的静态资源（bundle）
  2. 当使用 import $ from jquery 这个 es6 语法或者 less 语法时，浏览器都不能识别，需要一个构建工具将这些语法编译成浏览器能识别的代码。（就是将前端的一些编译小工具整合起来）
  3. 资源文件需要构建工具处理，先用打包起点（入口文件）开始打包，把每个依赖关系记录好形成依赖关系树状图，将类似 less、jQ 等形成 chunk（代码块），再进行一系列操作（类似将 less 编译成 css，es6 变成 es5 等）形成 bundle（静态资源）

# 1.2 五个核心概念

  1. Entry：指示哪个文件为入口开始打包，分析构建内部依赖图
  2. Output：指示打包后的资源 bundles 输出到哪里去，以及如何命名
  3. Loader：Loader 让 webpack 能够去处理那些非 JavaScript 文件（webpack 自身只理解 JavaScript）
  4. Plugins：插件可以用于执行范围更广的任务，插件的范围包括，从打包优化和压缩，一直到重新定义环境的变量
  5. Mode：模式指示 webpack 使用相应模式的配置：
选项描述特点
development会将 process.env.NODE_ENV 的值设为 development，启用 NamedChunksPlugin 和 NamedModulesPlugin能让代码本地调试的环境
production会将 process.enc.NODE_ENV 的值设为 production，启用 FlagDependencyUsagePlugin、FlagincludedChunksPlugin、ModuleConcatenationPlugin、NoEmitOnErrorsPlugin、OccurrenceOrderPlugin、SideEffectsFlagPlugin 和 UglifyJsPlugin能让代码优化上线运行的环境

# 2.webpack 基础

# 2.1 基本框架

  1. npm init -y：初始化项目
  2. npm i webpack webpack-cli -g：全局安装以便快捷使用指令，否则每次都要写 npx
  3. npm i webpack webpack-cli -D：安装开发依赖
  4. 新建 src 目录：源代码目录
  5. 新建 build 目录：通过打包处理后输出的目录
  6. 在 src 目录新建 index.js 作为入口文件
  7. index.js 代码如下：
/*
  1.运行指令：
    开发环境：webpack --entry ./src/index.js -o ./build --output-filename built.js --mode=development
    翻译：webpack 会以 ./src/index.js 为入口文件开始打包，打包后输出到 ./build/main.js，整体打包环境是开发环境
    生产环境：webpack --entry ./src/index.js -o ./build --output-filename built.js --mode=production
  2.结论：
    一、webpack 能处理 js 和 json 资源，不能处理 css/img 等其他资源
    二、生产环境比开发环境多一个压缩 js 代码
    三、生产环境和开发环境将 ES6 模块化编译成浏览器能识别的模块化
*/
import data from './data.json'
// import './index.css'

console.log(data)

function add(x, y) {
  return x + y
}

console.log(add(1, 2))

module: {

        'style-loader',
        'css-loader',
        // 将 less 文件编译成 css 文件

      ]

  ]

# 4.其他内容

# 4.1 output 中的 publicPath

publicPath：设置后，将改变 index.html 的内部引用路径。格式：域名 + publicPath + filename

举例：

  1. 设置 publicPath 为 '' 或不填：则 index.html 有：<script defer="" src="js/main.js"></script>
  2. 设置 publicPath 为 '/'：则 index.html 有：<script defer="" src="/js/main.js"></script>，但是这样的话，在使用 npm run build 就会报错，故设置为 ./ 才行

# 4.2 devServer 中的 publicPath

指定本地服务所在的目录，默认值为 /

举例：

  1. 设置 publicPath 为 /lg，则在浏览器中可以输入：localhost:8000/lg/index.html 访问到打包后的网页，与 4.1 引言的格式一致
  2. 但是如此设置，就会找不到 main.js，因为 output 里是从 / 中去找 js/main.js，所以 ouput 里的 publicPath 也得设置为 /lg
  3. 如果在 public 路径下有没有被打包的 uitls.js，且 index.html 引入了该包，打包后 utiils.js 没有被成功打包。故有属性 directory：打包之后的资源如果依赖了其他的资源，此时就告知去哪里找

代码：

devServer: {
  hot: 'only', // 模块热刷新
    static: {
      directory: path.join(__dirname, 'public'), // 静态资源的依赖在哪里找
      publicPath: '/lg', // 指定本地服务所在的目录
      watch: true
    }
}

output: {
  filename: 'js/main.js',
  path: resolve(__dirname, 'dist'), // 本地产出静态文件的存放位置
  publicPath: '/lg' // 改变 index.html 的内部引用路径
}
